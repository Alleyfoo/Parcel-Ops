### Switch the LLM showcase to ollama (commits e45c34e, f7e04b2, e0a1b9b, 95f37d1, b4b3f9a)

User feedback: "Hmm.. it does not look very functional. Lets just run it
locally on ollama. That atleast runs. I got ollama setup, but its not
accepting ollama server for a reason."

Root cause: the LLM client was hardcoded to the Gemini OpenAI-compat
endpoint. There was no `ollama` provider at all. The user already had
ollama running on 127.0.0.1:11434 (`ollama serve` was throwing a
port-in-use error precisely *because* ollama was already running).

**Files added or changed**
- `parcel_ops_llm/__init__.py` — added `OLLAMA_BASE_URL`,
  `DEFAULT_OLLAMA_MODELS`, `list_models()` (live `/v1/models` probe
  with curated fallback), and a provider-aware key check that lets
  ollama skip the API key requirement. Added `_normalize_hs_code()`
  to coerce LLM output to the canonical `NNNN.NN` form (smaller
  local models emit 8- or 10-digit codes by hallucinating subheadings).
- `llm_classifier.py` — provider-aware no-key placeholder; the check
  used to block ollama calls because the key was empty.
- `scripts/capture_llm_baseline.py` — added `--provider` (gemini|ollama),
  `--base-url`, `--no-pace`. Also fixed a long-standing bug: the script
  was calling `llm_classifier.classify_with_llm`, which always rebuilds
  cfg from streamlit session state — so even when the script built a
  valid ollama cfg, the inner call threw it away and built a default
  gemini cfg with no key. Now the script calls
  `parcel_ops_llm.classify_hs_code` directly with its own cfg.
- `app.py` — `render_llm_showcase()` now has a two-button provider
  toggle (Gemini / Ollama). Switching to ollama swaps the API key
  field for an Ollama base URL, populates the model dropdown from
  `/v1/models` (with a "↻ Refresh" button for after `ollama pull`),
  and enables the Run/Per-case buttons (no key required). The batch
  ETA caption and pacing are skipped for ollama (no rate limit).
  Added a `_ollama_server_reachable()` helper that surfaces a "start
  `ollama serve`" hint when the local server isn't responding.
- `llm_saved_results.json` — replaced the regex-only seed with a real
  ollama run (llama3.2:latest, all 12 cases, ~50s wall time).

see parse failures on a particular model, that's the first thing to
try removing. The parser already handles prose-around-JSON via

## What was completed this session

### Per-case Run button + paced batch for free-tier rate limits (commits 2ce2c72, 55505c7)

User feedback: free-tier Gemini is throttled to 5 RPM, so the "Run LLM on
all cases" batch kept hitting 429s and showing parse-failure noise. The
showcase is most useful when you can step through cases one at a time,
and the batch needs to behave well when the throttle kicks in.

**Files changed**
- `parcel_ops_llm/__init__.py` — added `RateLimitError(LLMError)` carrying
  a `retry_after` hint. `chat_completion()` detects HTTP 429 and parses
  the provider's `retry-after` value (fallback 13s, one slot past the
  12s free-tier window). `classify_hs_code()` catches it and returns
  `{"error": str, "rate_limited": True, "retry_after": N}`.
- `app.py` — `render_llm_showcase()`:
  - Paced the "Run LLM on all cases" loop to 13s/call (one slot past the
    5-RPM window), with a live ETA caption and a "remaining ~N min" hint
    in the button help text. On a 429 the batch halts cleanly and the
    remaining cases are marked with a stub `rate_limited: True` entry
    pointing the user at the per-case buttons.
  - Added a "Run LLM on this case" button at the top of each per-case
    expander. It calls the LLM for that one case, rebuilds the evald
    shape, patches `st.session_state["llm_results"][i-1]`, and
    `st.rerun()`s so the LLM column updates in place. Useful for
    retrying a throttled case or skipping ahead.

**Smoke test** (Playwright + HTTP)
- HTTP 200 on `streamlit run app.py` (port 8512)
- 5 tabs render; "LLM" tab clickable
- "LLM vs Regex" header found (count 2 — section + tab title)
- 24 "Run LLM on this case" buttons present (12 cases × DOM sub-nodes)
- First per-case button **disabled** when no API key is set ✓
- 0 console errors, 0 page errors
- First case expander text confirms correct layout:
  `Case 1: Plastic housing for electronic device · gold 8542.31 · ambiguous`
  followed by description, context, the per-case button, then the

**Important design choice**: I did NOT add a per-case exponential-backoff

metrics. If the user later asks for a silent retry, it's a small
addition to `classify_hs_code()`.

## Known issues

- The `initial_sidebar_state="collapsed"` page config means the LLM config
  panel lives inside the LLM tab (not the streamlit sidebar). This was
  intentional to keep the design system intact. If the user prefers a true
  sidebar, the config panel can be moved by changing `initial_sidebar_state`
  to `"expanded"` and wrapping the panel in `with st.sidebar:`.
- Free-tier Gemini has per-minute rate limits (5 RPM). The batch is now
  paced at 13s/call (~3 min for the full 12) and stops on the first 429.
  Per-case buttons let you retry throttled cases individually without
  re-running the whole batch.
- The probe cache lives in `st.session_state` only — closing the tab clears
  it. This is intentional (matches sql-editor pattern).
- The model list is hardcoded to the free-tier-friendly defaults. Newer
  models can be added by editing `DEFAULT_GEMINI_MODELS` in
  `parcel_ops_llm/__init__.py`.
- The 13s pacing is a *floor* — it does not retry on a 429 mid-batch. A
  429 at case 1 will skip cases 2-12 (they show "skipped" stubs). The
  user can then re-run the batch (or use the per-case buttons) after
  the free-tier window resets. A silent retry was deliberately not added
  so the user can see the throttle behaviour.

## Design decisions made

- `llm/` is a package (not a module) so future model clients (Claude, Ollama)
  can be added as siblings without restructuring.
- Config precedence matches sql-editor: `env > st.session_state >
  st.secrets > yaml`. No `config.yaml` is read in this project yet; the
  loader handles the missing case cleanly.
- The LLM config panel is a card inside the pipeline tab, not a sidebar
  widget, because the page config has `initial_sidebar_state="collapsed"`.
  The card uses the existing `.pipeline-card` design system class so it
  blends in with the rest of the tab.
- Regex rules are intentionally basic and ordered so that the **first match
  wins**. This mirrors what a real production regex pipeline would do and
  gives the LLM a fair target.
- Test cases are in a JSON file (not inlined) so they can be reused by a
  future CLI eval script, version-controlled separately, and edited without
  touching Python.
- The "expected_winner" field on each case is a UI hint, not a hard
  assertion. The gold code is the only ground truth.
- No new pip dependencies. The `google-generativeai` SDK would have been
  ~10MB; the stdlib urllib path is ~50 lines and works on Streamlit Cloud
  out of the box.
